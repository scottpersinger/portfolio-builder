# deploy-to-github

Publish a folder of static files (HTML/CSS/JS/images — e.g. a portfolio from
[[create_portfolio]]) to **GitHub Pages** and hand back a working public URL.

## When to use

The user has a self-contained static site on disk and wants it live on the web
for free: "deploy this", "host it on GitHub Pages", "give me a public link."

## Prerequisites

- `gh` is installed and authenticated (`gh auth status`). If not, tell the user
  to run `gh auth login` — **don't** try to log in for them.
- The site is **static** (no server-side code). GitHub Pages only serves static
  assets. If it needs a backend, say so and suggest Netlify/Vercel/Railway.
- **Publishing is public.** GitHub Pages sites are world-readable (even from a
  private repo). Confirm the user wants it public before pushing anything
  sensitive.

## Procedure

### 1. Get the files into a repo

- **Existing repo** (the files are already tracked): just make sure they're
  committed and pushed to the default branch. Note the `owner/repo` and branch.
- **New repo**: create one and push. From inside the site folder:

  ```bash
  git init -b main
  git add -A && git commit -m "Initial site"
  gh repo create <name> --public --source=. --remote=origin --push
  ```

Record `OWNER`, `REPO`, and `BRANCH` (usually `main`) — the rest uses them.

### 2. Add a `.nojekyll` file

Create an empty `.nojekyll` at the **published root** so GitHub serves your files
as-is instead of running them through Jekyll (which ignores files/folders
starting with `_` and can mangle static assets). Commit and push it.

```bash
touch .nojekyll && git add .nojekyll && git commit -m "Add .nojekyll" && git push
```

### 3. Decide what serves at the site root

Pages can only publish from the repo **root (`/`)** or a **`/docs`** folder — not
an arbitrary subfolder. So:

- Site files at the repo root → the Pages root is your site. Done.
- Site lives in a subfolder (e.g. `examples/`) and you want the **bare Pages URL**
  to land there → add a tiny root `index.html` that redirects:

  ```html
  <!DOCTYPE html><html><head><meta charset="UTF-8">
  <meta http-equiv="refresh" content="0; url=examples/">
  <link rel="canonical" href="examples/"></head>
  <body>Redirecting to <a href="examples/">the site</a>…</body></html>
  ```

  Otherwise the bare URL just renders the repo's README and the site sits at
  `.../<subfolder>/`.

### 4. Enable Pages via the API

```bash
gh api -X POST repos/OWNER/REPO/pages -f "source[branch]=BRANCH" -f "source[path]=/"
```

If Pages already exists (HTTP 409 / "already enabled"), update the source instead:

```bash
echo '{"source":{"branch":"BRANCH","path":"/"}}' | gh api -X PUT repos/OWNER/REPO/pages --input -
```

The response's `html_url` is where GitHub *thinks* the site will live — but see
step 6 before trusting it.

### 5. Wait for the build, then verify

Builds take ~30–90s. Poll until `built` (or `errored`), then curl the real URL:

```bash
# poll build status
for i in $(seq 1 10); do
  st=$(gh api repos/OWNER/REPO/pages/builds/latest --jq .status)
  echo "status=$st"; [ "$st" = built ] || [ "$st" = errored ] && break; sleep 12
done
# verify it actually serves (200, not a redirect into a 404)
curl -sL -o /dev/null -w "%{http_code}\n" https://OWNER.github.io/REPO/
```

Always **curl the URL and check for `200`** — don't declare success off the build
status alone. Follow redirects (`-L`) and inspect where they land.

### 6. The custom-domain redirect gotcha (check this if you get a 404)

If `https://OWNER.github.io/REPO/` **301-redirects to some other domain and 404s**,
the cause is almost always a **custom domain on the *user* Pages site**. When the
`OWNER.github.io` repo has a `CNAME` file (e.g. → `example.com`), GitHub redirects
**all** of that account's `github.io` traffic — including every project page — to
that domain. If that domain isn't actually served by GitHub Pages (it moved to
WordPress/Squarespace/etc.), every project page dead-ends in a 404.

Diagnose:

```bash
curl -sI https://OWNER.github.io/REPO/ | grep -i '^location'      # shows the redirect target
gh api repos/OWNER/OWNER.github.io/pages --jq .cname               # the offending custom domain
curl -sI https://<that-domain>/ | grep -i '^server'                # GitHub.com? or someone else?
```

Two fixes — **let the user choose, this touches domain/account config:**

- **The custom domain is stale** (the apex domain moved off GitHub Pages): remove
  it from the user-site repo. Safe — it only changes how GitHub routes
  `github.io`; it does **not** touch the domain's own DNS, so the live site there
  keeps working. Bonus: this un-breaks *every* project page on the account.

  ```bash
  echo '{"cname": null}' | gh api -X PUT repos/OWNER/OWNER.github.io/pages --input -
  ```

- **The custom domain is still in use**: leave the user site alone and instead give
  **this** repo its own subdomain. Add a `CNAME` file with e.g.
  `portfolio.example.com`, set it as this repo's Pages domain, and have the user
  add a DNS `CNAME` record `portfolio → OWNER.github.io`. A repo-level custom
  domain overrides the user-site redirect for this repo.

After the fix, re-run step 5's curl until it returns `200` (CDN propagation can
take a minute or two).

### 7. Report

Give the user the **verified** live URL(s). If you added a subfolder redirect,
mention both the bare URL and the subfolder URL. Note that the site is public and
how to update it (push to the branch — Pages rebuilds automatically).

## Notes

- Updates are automatic: every push to the published branch triggers a rebuild.
- HTTPS is provided free on `*.github.io`; for a custom domain, enable
  "Enforce HTTPS" in Pages settings once the cert is issued.
- Relative asset paths (`images/…`, `./style.css`) work under the `/REPO/`
  path prefix; **absolute** paths (`/images/…`) break on project pages — prefer
  relative.
